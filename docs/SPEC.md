# 데일리 TODO 리워드 앱 — 기능 명세서 (SPEC)

> 이 문서는 프로젝트의 "진실의 원천(single source of truth)"입니다.
> 모든 기능 규칙을 사람이 읽을 수 있게 상세히 적어둡니다.
> CLAUDE.md는 이 문서에서 핵심만 압축해 뽑아낸 요약본입니다.
> 확정되지 않은 항목은 `[미정]` 으로 표시합니다.

---

## 0. 한 줄 요약

사용자가 매일 TODO 미션을 완료하며 포인트를 모으고, 그 포인트로
(A) 아바타를 꾸미거나 (B) 코인머니로 전환해 출금하는, 재미 중심의 습관 형성 앱.

---

## 1. 사용자 구조와 권한

- **총 2인 구조**: 관리자 1명 + 사용자 1명. (사용자는 본인 또는 가족 등)
- 계정 인증은 단순하게 유지한다 (사용자 수가 고정이므로 복잡한 회원가입 흐름 불필요).

### 권한 매트릭스

| 기능 | 사용자 | 관리자 |
|---|:---:|:---:|
| TODO 실행 여부 체크 (완료 토글) | O | - |
| TODO 우선순위 정렬 (드래그앤드롭, 순서만 변경) | O | - |
| 포인트 샵(A상점) 이용 | O | - |
| 코인머니 전환 조회 | O | - |
| 캘린더 조회 | O | O |
| 뱃지 화면 조회 | O | O |
| TODO 입력 / 삭제 / 수정 | - | O |
| 루틴 설정 | - | O (기본) `[미정: 사용자도 가능하게 할지]` |
| 출금 수동 처리 (계좌 지급 + 코인머니 차감) | - | O |
| 연속 보너스 리셋 주기 설정 | - | O |

> 핵심 원칙: 사용자는 "실행과 소비", 관리자는 "설정과 관리"를 담당한다.

## 2. 포인트 시스템 (앱의 심장)

### 2.1 두 종류의 포인트

- **A포인트**: 캐릭터 스토어(A상점) 전용.
- **B포인트**: 적립 통장(B상점) 전용.
- **적립은 함께, 차감은 따로**:
  - 미션을 완료하면 A포인트와 B포인트가 **항상 정확히 같은 양** 동시에 적립된다.
  - A상점에서 소비하면 **A포인트만** 차감된다. B포인트는 영향 없음.
  - B상점에서 전환하면 **B포인트만** 차감된다. A포인트는 영향 없음.
  - 즉, 두 개의 독립된 지갑에 같은 금액을 동시에 넣어주되, 꺼내 쓰는 것은 완전히 별개다.

### 2.2 적립 규칙

| 항목 | 포인트 | 조건 |
|---|---|---|
| 미션 1개 완료 | +5 | TODO 하나를 완료할 때마다 (A·B 각각 +5) |
| 완주 보너스 | +10 | 그날의 모든 TODO를 완료했을 때 |
| 연속 보너스 | +10 × (N−1) | N일 연속 완주 시, N일째에 지급 |

- **연속 보너스 상세** (완주한 날 기준으로 누적):

| 연속 일수 N | 완주 보너스 | 연속 보너스 | 그날 보너스 합계 |
|---|---|---|---|
| 1일째 | +10 | +0 | +10 |
| 2일째 | +10 | +10 | +20 |
| 3일째 | +10 | +20 | +30 |
| 4일째 | +10 | +30 | +40 |
| N일째 | +10 | +10 × (N−1) | 10 + 10×(N−1) |

- **상한(cap)**: 기본은 **무한정**. 상한은 두지 않는다.
- **연속 보너스 리셋**: 관리자가 "며칠마다 연속 보너스를 리셋할지" 설정할 수 있는 메뉴를 둔다.
  - 예: 리셋 주기를 7일로 설정하면, 7일 연속 후 다음 완주는 다시 1일째부터 시작.
  - 기본값: **리셋 안 함** (설정하지 않으면 연속은 계속 무한정 누적).

### 2.3 연속(streak) 판정 규칙

- "완주"란 그날 배정된 모든 TODO를 완료한 상태를 뜻한다.
- 하루라도 완주하지 못하면 연속은 0으로 리셋되고, 다음 완주 시 1일째부터 다시 센다.
- **TODO가 하나도 없는 날의 처리**: 연속을 **끊는다** (완주할 TODO가 없으므로 완주 불성립).
  - 안전장치: 해당일에 TODO가 등록되어 있지 않으면, 관리자가 로그인할 때
    웹/앱에 "오늘(해당일) TODO가 등록되지 않았습니다" 푸시 알림을 띄운다.
    (관리자가 미리 등록해 사용자의 연속이 억울하게 끊기는 것을 방지)
- **하루 경계 기준 시각**: 관리자가 설정할 수 있다. 기본값은 **밤 12시(자정)**.
  - 이 값이 하루의 시작/끝을 정한다. 예: 아침 9시로 설정하면 하루는
    "오전 9시 ~ 다음 날 오전 9시"로 계산된다.
  - 이 경계 시각은 연속 판정, 완주 판정, 일요일 결산 시각에 **일관되게** 적용된다.
  - 타임존: `[미정: 고정 KST로 둘지]`

### 2.4 포인트 지급 시 시각 효과

- 포인트가 적립될 때 화면에 "팡 터지는" 재미있는 시각 효과를 반드시 넣는다.
  (파티클/컨페티, 숫자가 튀어오르는 애니메이션 등)
- 미션 단건 완료 시와 완주/연속 보너스 시의 효과 강도를 다르게 하여 성취감 차등. `[미정: 구체 연출]`

## 3. A상점 — 캐릭터 스토어 + 아바타

### 3.1 아바타 기본

- 사용자에게 기본 아바타가 주어진다.
- 아바타는 여러 "레이어(부품)"를 겹쳐 표현한다 → 아이템 교체가 쉬운 구조로 설계.

### 3.2 아이템 카테고리

상의, 하의, 원피스, 셋업, 신발, 양말, 구두, 선글라스, 액세서리, 가방,
배경, 펫(종류 다양), 풍선 등 잡동사니, 헤어스타일 등.

- 아이템마다 A포인트 가격이 **차등** 책정된다.
- 구매한 아이템은 내 옷장에 **카테고리별**로 저장된다.
- 옷장에서 아이템을 **착용/해제(탈착)** 할 수 있다.

### 3.3 아바타 룸

- 내 아바타를 꾸미는 공간.
- 현재 착장 상태의 아바타를 **캡처하여 이미지로 저장**할 수 있다.

### 3.4 그래픽 조달 전략

- 개발 초기: 무료 오픈소스 아바타 시스템(DiceBear, Avataaars 등)으로 레이어/탈착 로직 완성.
- 다듬기 단계: 통일된 스타일의 무료/구매 에셋 팩 또는 Claude Design 산출물로 교체.
- 핵심: 처음부터 "레이어 겹치기" 방식으로 설계해 그래픽만 나중에 교체 가능하게.

## 4. B상점 — 적립 통장 + 코인머니

- B상점은 소비형 상점이 아니라 "적립 통장"이다.
- **매주 일요일 자동 캡전환**: 그 주 결산 시점의 B포인트 잔액을, 전환 규칙에 맞는
  최대 금액까지 코인머니로 자동 전환하고, 남는 자투리는 다음 주로 이월한다.
  - **결산 시각은 하루 경계 시각을 따른다.** 기본값은 일요일 밤 12시(자정).
    경계 시각이 바뀌면 결산 시각도 함께 움직인다. 정확히는 "일요일이라는 하루가
    끝나는 경계"에 결산한다. (예: 경계가 오전 9시면, 일요일이 끝나 월요일로
    넘어가는 오전 9시에 결산)
  - **사용자 안내(툴팁)**: 경계 시각 설정에 따라 결산 시각을 툴팁으로 안내한다.
    예: 경계가 오전 9시면 "결산은 일요일 아침 9시입니다".
  - 자동 실행은 크론 잡(예약 작업)으로 처리.
  - **중복 실행 방지(멱등성)**: 결산 기록에 "연도-주차" 유일 키를 걸어,
    같은 주의 결산이 두 번 실행되어도 코인머니가 이중 지급되지 않게 한다.
- 코인머니로 전환된 금액은 **출금 가능**하다.

### 4.0 전환 규칙 (자동 캡전환)

- **전환 비율**: B포인트 1점 = 코인머니 1원 (1:1).
- **최소 전환 금액**: 500원. 잔액이 500 미만이면 전환하지 않고 전액 이월.
- **전환 단위**: 500원 이상이면 100원 단위로만 전환. 100 단위로 안 떨어지는
  자투리(끝 두 자리, 0~99)는 다음 주로 이월.
- **계산식**: `잔액 >= 500` 이면 `전환액 = 잔액 - (잔액 % 100)`, `이월 = 잔액 % 100`.
  - (`%`는 나머지 연산. 예: 730 % 100 = 30)
- **이월 처리**: 이월된 자투리는 B포인트로 그대로 남아 다음 주에 새로 쌓인 B포인트와
  합산된다. 별도 이월 계정을 두지 않는다.

| 결산 시 B포인트 잔액 | 전환되는 코인머니 | 이월(잔여 B포인트) |
|---|---|---|
| 470 | 0원 | 470 |
| 499 | 0원 | 499 |
| 500 | 500원 | 0 |
| 730 | 700원 | 30 |
| 1,250 | 1,200원 | 50 |

> 주의: 이 전환은 B포인트에서만 일어난다. A포인트는 전혀 관여하지 않는다.

### 4.1 출금 프로세스

1. 사용자 앱: 코인머니 잔액 확인 및 전환 조회.
2. 관리자 앱: 관리자가 **수동으로 출금 처리**한다.
   - 관리자가 직접 계좌로 지급.
   - 지급한 만큼 사용자 앱에서 코인머니를 차감.
- 자동 계좌 이체는 없다. 관리자의 수동 처리가 핵심.

## 5. 캘린더

- 이전 날짜 / 오늘 / 특정 날짜를 클릭하면 그날의 TODO 목록을 볼 수 있다.
- **디폴트 화면은 심플하게**: 오늘의 TODO 목록과, 각 항목의 완료 여부 토글(on/off)만.
- 조회 전용 (사용자는 캘린더에서 TODO를 편집하지 않는다).

## 6. TODO 기능

### 6.1 사용자가 할 수 있는 것

- 완료 여부 체크(토글 on/off).
- 우선순위 정렬: 드래그앤드롭으로 **순서만** 변경 (내용 편집 아님).

### 6.1.1 토글 규칙 (포인트 정합성의 핵심)

- 사용자는 **"오늘"(하루 경계 시각 기준)의 TODO만** 토글할 수 있다.
  과거/미래 날짜의 TODO는 토글 불가 (캘린더는 조회 전용).
- **토글 ON(완료)**: A·B 포인트 각각 +5. 이로 인해 그날 전체가 완주 상태가 되면
  완주 보너스(+10)와 연속 보너스(+10×(N−1))를 즉시 지급하고 연속 일수를 갱신.
- **토글 OFF(완료 취소)**: 해당 미션의 +5를 A·B 각각 회수.
  이로 인해 완주 상태가 깨지면, 그날 지급했던 완주·연속 보너스도 전액 회수하고
  연속 일수도 완주 이전 상태로 되돌린다.
- 목적: "체크→해제" 반복으로 포인트를 무한 적립하는 악용을 원천 차단.
  모든 지급·회수는 point_transactions에 기록되어 추적 가능해야 한다.

### 6.2 관리자가 할 수 있는 것

- 사용자의 TODO 입력 / 삭제 / 수정.

### 6.3 실시간 추천 검색어 (자동완성)

- 관리자가 새 TODO를 입력할 때, 타이핑에 따라 과거 유사 TODO를 실시간으로 추천.
- 동작: 타이핑할 때마다(디바운스 적용) 백엔드에 조회 → 과거 유사 케이스 검색 → 아래에 추천어 표시.
- 추천어 클릭 시 입력창에 자동 완성.
- 초기 구현: 단순 문자열 포함/전방 일치 검색.
- 고도화(선택): 의미 기반 임베딩 검색으로 확장 가능.

## 7. 루틴 (반복 TODO 자동 생성)

- 한 가지 업무에 대해 반복 규칙을 설정하면, 규칙에 따라 TODO가 자동으로 뜬다.
- 설정 항목:
  - 시작일 ~ 종료일
  - 반복 유형: 매일 / 요일별(특정 요일 선택) / 공휴일 제외 / 등
- 목적: 규칙적인 TODO를 매번 수동 입력하지 않아도 되게 한다.
- 공휴일 데이터 출처: `[미정: 공휴일 API 또는 수동 등록]`

## 8. 뱃지 (동기부여)

- TODO 업적에 따라 다양한 뱃지를 모으는 뱃지 화면. 사용자·관리자 모두 조회 가능.
- 초기 승리 경험을 위해 "지키기 쉬운" 뱃지 위주로 20종.

## 9. 안드로이드 위젯

- 홈 화면 위젯으로 오늘의 TODO를 표시하고 완료 토글 가능.
- 네이티브 모듈 필요 → 개발 우선순위 최하위 (앱 본체 완성 후 얹는 보너스).

## 10. 클라이언트 (앱 / 태블릿 / 웹)

- **React Native + Expo** 단일 코드베이스로 안드로이드·iOS·웹을 커버.
- **태블릿**은 별도 앱이 아니라 큰 화면 대응 레이아웃으로 처리.

## 11. 백엔드 / 인프라

- **백엔드**: FastAPI (Python).
- **DB**: PostgreSQL.
- **로컬 개발**: Docker Compose로 FastAPI + PostgreSQL 구동.
- **클라우드 전환(무료 우선)**:
  - 백엔드: Render 무료 웹 서비스 (콜드 스타트 감수).
  - DB: Supabase 또는 Neon 무료 Postgres (백엔드와 분리해 데이터 보존).
  - 웹 프론트: Vercel 또는 Netlify 무료.
  - 파일 저장(아바타 이미지): Supabase Storage 또는 Cloudflare R2.
  - 크론 잡(일요일 결산): Render Cron 또는 Supabase pg_cron.
