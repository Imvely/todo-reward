# /morning — 하루 시작 프로토콜

사용자 개입을 최소화한다. 아래 단계를 **순서대로, 빠짐없이** 수행하고, 질문은 정말 사용자만 결정할 수 있는 사항(SPEC의 `[미정]` 등)에 한정한다.

## 자동 수집 컨텍스트

- 현재 브랜치와 변경: !`git status --short --branch`
- 최근 커밋 10개: !`git log --oneline -10`

## 수행 단계

1. **어제 정리 확인**: `docs/worklog/`에서 가장 최근 데일리 로그를 읽는다.
   - "내일 할 일" 섹션이 오늘의 1차 후보 작업이다.
   - 어제 로그가 없으면(예: /evening을 안 하고 퇴근) 위의 git log와 diff로 어제 한 일을 역으로 복원해 요약한다.
2. **백로그 확인**: `docs/worklog/BACKLOG.md`를 읽고 미완료 항목(`- [ ]`) 전체를 파악한다.
   완료됐는데 체크 안 된 항목이 보이면 체크한다.
3. **오늘 계획 수립**: 어제의 "내일 할 일" + BACKLOG 우선순위 + `$ARGUMENTS`(있으면 최우선 반영)를 종합해
   오늘 할 일 1~4개를 고른다. 로드맵 순서(SPEC §13)와 의존성을 존중한다.
4. **오늘 워크로그 생성**: `docs/worklog/TEMPLATE.md` 형식으로 `docs/worklog/YYYY-MM-DD.md`(오늘 날짜)를 생성하고
   "오늘 목표" 섹션에 계획을 체크박스로 기록한다. 이미 있으면 목표만 갱신한다.
5. **브리핑 출력**: 사용자에게 3~6줄로 보고한다 — ① 어제 한 일 요약 ② 오늘 할 일 목록(우선순위 순) ③ 첫 작업이 무엇인지.
6. **바로 작업 시작**: 승인을 기다리지 말고 1순위 작업을 즉시 진행한다.
   단, CLAUDE.md의 작업 절차를 반드시 지킨다:
   - 구현 전 TECH_DESIGN에서 해당 절 번호를 찾아 읽고 계획에 인용한다. 해당 절이 없으면 설계 제안 후 승인 대기.
   - SPEC `[미정]` 항목은 임의로 정하지 말고 질문 목록으로 모아 브리핑에 포함한다.
   - 포인트/코인/연속 로직을 건드리면 TECH_DESIGN §6 테스트를 실행해 통과를 확인한다.
7. 작업 진행 중에는 오늘 워크로그의 "진행 기록"에 중요한 결정·전환점을 짧게 남긴다.

## 규칙

- 모든 문서는 `.md`로만 저장한다 (.txt 금지 — 회사 정책).
- 날짜는 시스템 오늘 날짜(세션 컨텍스트에 주입된 값)를 쓴다. 임의 추정 금지.
